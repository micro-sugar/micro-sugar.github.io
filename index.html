私人NPM伺服器

發表於 2019-11-25 |

當我們進行前端開發時，總是避免不了的大量使用 npm 指令安裝套件；不過實際在工作時常會遇到一種狀況就是客戶的環境連不到 internet，這邊會介紹如何使用 Verdaccio 架設私人 NPM 伺服器的方式解決這個問題。

Verdaccio 功能

  • 快取npmjs.org的registry
  • 覆蓋公共套件

    如果要使用某個第三方軟件包的修改版本（例如，您發現了一個錯誤，但維護者尚未接受PR請求），則可以使用相同的名稱在本地發布該版本。

  • 建立私人的npm伺服器

環境需求

  • NodeJs
  • IIS
  • iisnode-full-v0.2.26
  • Web Platform Installer
  • URL Rewrite

安裝 Verdaccio

  • 建立資料夾給 verdaccio 使用

  • verdaccio 資料夾增加package.json檔案

  • verdaccio 資料夾增加start.js檔案
  • 建立 verdaccio 設定檔config.yaml
  • 安裝 verdaccio 相依套件

  • 新增verdaccio應用程式集區

  • verdaccio 資料夾增加web.config檔案

  • 確定 IIS 開啟 CGI 功能

  • verdaccio 資料夾增加IIS AppPool\verdaccio寫入權限

  • 新增 verdaccio 站台

  • 確認處理常式對應裡的iisnode是否有勾指令碼\執行權限

  • 開啟瀏覽器確定是否執行成功

    到此處verdaccio環境及安裝基本上已經完成

下載專案會使用到的 node package

由於 verdaccio 有 cache 的功能，所以只要先將 npm 的 registry 指到 verdaccio 後再安裝專案需要的 node package 後，專案所需的 package 就會被 Cache 進 verdaccio 了

  • 修改電腦 npm registry
  • 於前端專案執行npm i

  • 此時可以看到C:\verdaccio\storage下會有專案會用到的Node Package

  • 到此時 Verdaccio server 已經將專案會用到的package快取起來

    • 如將網路斷線，但開發人員的電腦還是可以連到 Verdaccio server 的狀況下，將仍可透過 Verdaccio server 進行npm install
    • 如有需要也可以再有internet環境架設且下載完專案需要用的node package後，進內網且將環境建置後拿外網storage資料夾覆蓋後即可在斷網環境下完成建置

Windows下使用ssh連結Bitbucket(Git)

發表於 2018-11-08 | 分類於 Git |

前陣子在使用Git的時候遇到一個很麻煩的狀況
就是使用Git時，不管是push還是pull，都一直要我打密碼，有時候光pull一次就要打好幾次密碼
這是由於Git可以使用ssh或是https的方式進行連結，如果選擇到了https的方式，又沒特別去處理的話就會有需要一直打密碼的狀況出現
如果不想一直輸入密碼有幾種方式可以選

  • 直接將帳號密碼加到url內(由於可能會被proxy之類的Log記住，故強烈不推薦!!)

  • 使用.git-credentials儲存帳號密碼
    這個方法會在%userprofile%下產生一個.git-credentials檔案，並在裡面明碼儲存

  • 使用ssh方式連線

因為使用Linux或MacOS的人應該都對ssh的部分很熟悉
所以，這篇是要教大家在Winsows下使用ssh的方式連接Git(Bitbucket)

環境需求

本機

  • Git

線上服務

線上的Git repository都可以，這邊是用Bitbucket當作範例

  • Bitbucket

製作金鑰

開啟Git Bash

在Windows安裝完Git，就會有Git Bash可以用
打開開始後直接打Git Bash就可以看到了

確認是否已經有ssh金鑰

這邊有兩種做法
可以選擇在檔案總管看或是在Git Bash上打指令確認

檔案總管

直接在檔案總管輸入%userprofile%\.ssh看看這個資料夾是否存在

Git Bash

在Git Bash上cd到~/.ssh看看該資料夾是否存在

如果回傳No such file or directory代表目前ssh金鑰不存在

製作ssh金鑰

如果不存在，就可以開始準備製作ssh金鑰
一樣是在Git Bash使用ssh-keygen產生金鑰

其中雙引號內的e-mail請替換為自己的e-mail
輸入後會被問三個問題(如下列)，如無特殊需求，都直接按Enter就可以

如看到類似下面這樣的文字，代表已經完成

到這邊，會在%userprofile%\.ssh建立2個檔案

  • id_rsa:金鑰(Private key)
  • id_rsa.pub:公開金鑰(Public key)

Bitbucket加入public-key

首先先將id_rsa.pub用記事本打開，並將裡面的文字複製起來

登入Bitbucket以後依下圖選擇Add key


選擇後會出現下圖畫面
在Key的欄位將id_rsa.pub的文字貼上
Label欄位可以隨便輸入，方便自己辨識哪個key或是哪台電腦就行


輸入完再按右下方的Add key
這樣就完成了!

VSTS於Build和Release時會用到的位置變數

發表於 2018-09-10 | 分類於 VSTS |

剛開始用VSTS的時候常被他的Build或Release的位置變數搞混
所以這篇文章記錄一下自己常用的變數

下面有些資料夾可以用不同變數到同一個位置，我就放在一起了

需要完整的變數表可以從備註/相關資料連結去Microsoft的網站上查詢

Build

VSTS或TFS的Agent在執行Build的時候主要是在4個資料夾內進行動作

BuildDirectory

For example: c:\agent_work\1

  • 給該pipeline的資料夾
  • 有些套件會在這個地方增加資料夾給套件自己使用

ArtifactStagingDirectory

For example: c:\agent_work\1\a

  • Artifacts是您希望構建生成的文件(例如，您有一個C＃或C ++ .NET Windows應用程序的.DLL和.EXE可執行文件和.PDB符號文件。)
  • 換個方式說就是待部屬的資料夾
  • 此資料夾在每次新的Build之前會自動清除此目錄，因此不必自己清理它。

BinariesDirectory

For example: c:\agent_work\1\b

  • 預設不會自動清除，如須清除請於Get sources的Clean選True，Clean options選擇Sources and output directory(此選項會刪除BinariesDirectory，ArtifactStagingDirectory，TestResultsDirectory)
  • 由組建代理程式編譯的檔案，例如已編譯的應用程式檔案。

SourcesDirectory

For example: c:\agent_work\1\s

  • 由組建代理程式讀取的檔案，例如原始程式檔。
  • VSTS的Get sources會將Source下載至子資料夾
  • 預設是不會清除的

TestResults

For example: c:\agent_work\1\TestResults

  • 由組建代理程式執行的任何測試產生的檔案。

Release

Release的部分基本上只會產生一個資料夾

ArtifactsDirectory

For example: C:\agent_work\r1\a

在執行階段利用command line或是powershell可以做另外的操控(如上所述，部分擴充套件也會建立自己的資料夾喔)
